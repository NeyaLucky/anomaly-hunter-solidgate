# Transaction Anomaly Detection

Unsupervised anomaly detection on 1,000,000 payment transactions from a single merchant's 2025 activity (Solidgate hackathon task). No labeled data — anomalies are found through statistical deviations, broken business invariants, and artificial patterns, then cross-checked with an unsupervised model.

## Task

Label every transaction as `1` (anomalous) or `0` (normal) and explain *why*. Scored by F1 under heavy class imbalance, so the priority is precision: every flagged row needs a concrete, defensible reason, not a statistical hunch.

## Approach

1. **Data quality pass** — integrity checks (duplicate keys, timestamp ordering, missing-value logic) to separate real defects from structural nulls before touching anomaly hunting.
2. **Rule-based investigation** — each lead is isolated with a boolean filter, profiled, and only trusted once it shows a concentrated fingerprint (a fixed time window, a single provider, a constant magnitude) rather than being explainable by ordinary customer/business variance.
3. **ML cross-check** — an `IsolationForest` run on generic engineered features (latency, currency-normalized amount, fail flag), used to validate the rule-based findings and probe for anything missed.

## Findings

| Cluster | Rows | Signature | Nature |
|---|---|---|---|
| `bank_id == 777` | 635 | Constant 5.0s processing latency (zero variance), 100% fail, all created on one calendar day, a 3-digit ID unlike every other `bank_id` | Synthetically inserted block |
| `psp_gamma` latency incident | 5,433 | 1–2h processing delay, gradual degrade/recover shape over 6 days, normal fail rate | Real provider-side outage — **not** flagged |
| `psp_beta` over-refund | 2,691 | `refunded_amount` exceeds `amount` by a flat +10 in every currency, confined to a 5-day window, some on failed transactions | Synthetic / corrupted financial data |
| Refund on a failed transaction | 1,299 | `has_refund` on a `status == fail` row (logically impossible — a failed charge collects no money); 94% at exactly half the transaction amount | Synthetic, present all year |

Also investigated and ruled out as ordinary variance rather than anomalies:
- `ip_country`/`bin_country` mismatches — explained by customers using home-issued cards while traveling
- per-category `amount` outliers — a legitimate, currency-converted subscription price ladder
- `psp_alpha`'s persistently higher fail rate — systemic across the full year, too broad to be a point anomaly

The `IsolationForest` cross-check independently rediscovers 3 of the 4 clusters from generic features alone; it structurally cannot see `bank_id == 777`, since that cluster's signature is zero variance *within* a group rather than any single row looking unusual — a collective, not a point, anomaly.

**Final label:** the union of the 3 synthetic clusters (the real `psp_gamma` outage is deliberately excluded) — 4,559 of 1,000,000 rows (0.46%).
